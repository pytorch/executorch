# Gemma4 ExecuTorch — CUDA-machine handover

**Source branch:** `younghan/gemma4-xnnpack-fp32` (despite the name, the work transfers cleanly to CUDA)
**Last validated:** 2026-04-22 on CPU+XNNPACK (FP32 v11 and 8da4w v11_q both passing 9/10 tests)
**This doc:** what to do on the CUDA box to repeat the same end-to-end success on GPU.

---

## What's already done (don't redo)

All Gemma4 modeling and runner code works as-is. You inherit:

- A 5-method multimodal `.pte` schema (text_decoder + token_embedding + vision_encoder + audio_preprocessor + audio_encoder) compatible with the standard `MultimodalRunner`.
- A C++ runner (`main.cpp` → `gemma4_runner`) that already auto-handles CUDA via the existing `--data_path` flag for the AOTI blob.
- A working CMake target `make gemma4-cuda` that already links `aoti_cuda_backend`.
- An export script (`export_gemma4_multimodal.py`) that takes `--backend xnnpack` today and accepts `--qmode 8da4w --group-size 32` for quantization.
- Layer-by-layer parity tests vs HF (`tests/test_parity.py`) and KV-cache + multimodal-wrapper bit-exact tests (`tests/test_kvcache_decode.py`, `tests/test_textdec_wrapper.py`).
- The PLI bug-fix that made multimodal work end-to-end. It is in source at `examples/models/llama/llama_transformer.py` (Transformer.forward reads `attn_options["pli_token_ids"]` when `tokens` is None). Committed in `e485049872`.

What is **not** yet done on this branch:

1. CUDA backend wiring inside `export_gemma4_multimodal.py` (XNNPACK-only today). Adding it is ~20 lines following the dinov2 pattern (see below).
2. Validation of CUDA execution — neither inference correctness nor performance has been measured for Gemma4.

## CRITICAL gotcha: keep the installed `executorch` package in sync with the source tree

Symptom if you skip this: multimodal generation degenerates to "Paris-Nicolas-Nicolas-Paris-Nicolas-..." within 5–10 tokens. PLI is silently zeroed out. Two days were lost on the wrong diagnosis ("sequential KV-cache prefill") before realizing the installed package was old. Don't repeat that.

After cloning the branch:

```bash
cd ~/executorch
./install_executorch.sh        # first time
# OR
pip install -e . --no-build-isolation   # subsequent
```

Then verify:

```bash
diff -q examples/models/llama/llama_transformer.py \
        $(python -c 'import executorch.examples.models.llama.llama_transformer as t; print(t.__file__)')
# must print nothing — files identical
```

If they differ, `cp` the source over the installed copy (or reinstall). The export pipeline imports the **installed** `executorch.examples.models.llama.llama_transformer`, not the source tree.

A bit-exact regression test is provided:

```bash
PYTHONPATH=$PWD python examples/models/gemma4/tests/test_textdec_wrapper.py
# Expected: "max_diff vs ref: 0.0000e+00" and clean "Paris**.<turn|>" generation
```

If `max_diff` is anything else, your installed package is stale.

## Step 1 — Build the CUDA runner

```bash
make gemma4-cuda
```

That target is already wired in `Makefile` line 400. It runs `cmake --workflow --preset gemma4-cuda` which sets `EXECUTORCH_BUILD_CUDA=ON` and links `aoti_cuda_backend`. The runner binary lands at `cmake-out/examples/models/gemma4/gemma4_runner`. The runner accepts `--data_path` for the AOTI `.ptd` blob.

If the build complains, check `examples/models/dinov2/CMakeLists.txt` for the canonical CUDA wiring — it's the simplest reference. `examples/models/voxtral/CMakeLists.txt` is the closest multimodal-runner analogue.

## Step 2 — Add CUDA backend to the export script

Today `export_gemma4_multimodal.py:_partitioners()` only knows `xnnpack` and `portable`. Add a `cuda` branch by mirroring `examples/models/dinov2/export_dinov2.py:60–73` and `examples/cuda/scripts/export.py:14–95`. Sketch:

```python
def _partitioners(programs: dict, backend: str):
    if backend == "xnnpack":
        from executorch.backends.xnnpack.partition.xnnpack_partitioner import (
            XnnpackDynamicallyQuantizedPartitioner, XnnpackPartitioner,
        )
        return {k: [XnnpackDynamicallyQuantizedPartitioner(), XnnpackPartitioner()]
                for k in programs}
    if backend == "cuda":
        from executorch.backends.cuda.cuda_backend import CudaBackend
        from executorch.backends.cuda.cuda_partitioner import CudaPartitioner
        # CudaBackend wants a per-method compile spec keyed by method name.
        return {k: [CudaPartitioner([CudaBackend.generate_method_name_compile_spec(k)])]
                for k in programs}
    return []
```

Then add `"cuda"` to the `--backend` choices in the argparse block (line 376). The CUDA backend generates a separate AOTI blob that needs to be saved alongside the `.pte`. Look at how dinov2/voxtral persist `aoti_cuda_blob.ptd` — that's the runner's `--data_path` argument.

Decompositions: `dinov2` runs `conv1d_to_conv2d` decomposition before lowering. The Gemma4 audio encoder uses depthwise `conv1d`, so this decomposition will likely be needed for `audio_encoder`. Apply it conditionally in `lower_all` for the audio-encoder program.

## Step 3 — Re-export

```bash
PYTHONPATH=$PWD python examples/models/gemma4/export_gemma4_multimodal.py \
  --hf-model ~/models/gemma-4-E2B-it \
  --et-checkpoint ~/models/gemma-4-E2B-it/model_et.pth \
  --output /tmp/gemma4_multimodal_v11_cuda.pte \
  --backend cuda --max-seq-len 1024 --audio-frames 1976
```

For BF16 (CUDA-friendly), follow dinov2's `dtype=torch.bfloat16` pattern. Quantized:
add `--qmode 8da4w --group-size 32`. Note that 8da4w on CUDA may need the
`pt2e_quantize` path instead of source-transform `qmode`; check what gemma3-cuda
uses (`examples/models/gemma3/`).

Long-running command — the XNNPACK FP32 export takes ~30–40 min; expect CUDA to
be similar. **Don't `| tail`** during the export; tee to a log file or run
unwrapped. Poll with `grep "Saved\|Lowering\|Error" /tmp/gemma4_mm_export_v11_cuda.log`.

```bash

MODEL="/tmp/gemma4_multimodal_v11_cuda.pte"
```

Or run individual tests with `--data_path` if CUDA needs the AOTI blob:

```bash

MODEL=/tmp/gemma4_multimodal_v11_cuda.pte
BLOB=/tmp/aoti_cuda_blob.ptd       # whatever the export wrote
TOK=~/models/gemma-4-E2B-it/tokenizer.json

$RUNNER --model_path $MODEL --data_path $BLOB --tokenizer_path $TOK \
        --prompt "What is the capital of France?" --seq_len 30
$RUNNER --model_path $MODEL --data_path $BLOB --tokenizer_path $TOK \
        --image_path image.jpg --prompt "Describe this image." --seq_len 60
$RUNNER --model_path $MODEL --data_path $BLOB --tokenizer_path $TOK \
        --audio_path obama_short20.wav --prompt "Transcribe this audio." --seq_len 80
```

Acceptance bar — outputs should look like:

| Test | Expected |
|------|----------|
| Text  | `The capital of France is **Paris**.<turn\|>` |
| Image | A coherent description that mentions "red", "strawberry", and stops cleanly |
| Audio | Real transcription of the Obama speech ("This week I traveled to Chicago...") |

Anything resembling "Paris-Nicolas-Nicolas" means PLI is broken — go back to "CRITICAL gotcha" above and verify the installed-package sync.

## Background: what the working .pte looks like

```
/tmp/gemma4_multimodal_v11.pte    21 GB  XNNPACK FP32  (validated, 9/10 pass)
/tmp/gemma4_multimodal_v11_q.pte  13 GB  XNNPACK 8da4w (validated, 9/10 pass, +25% decode)
```

CPU baseline numbers to beat on GPU (XNNPACK 8da4w, single .pte all-modalities):

| Modality | Prefill tok/s | Decode tok/s |
|----------|---------------|--------------|
| Text     | 108–122       | 15.9–16.5    |
| Image    | 100–102       | 14.1–14.6    |
| Audio    | 311           | 13.7         |

Method ABI (do not change without re-aligning runner):

```
text_decoder      (embeds[1,S,1536], cache_pos[1], pli_token_ids[1,S]) → logits[1,262144]
token_embedding   (token_ids[1,S]) → (1,S,1536)        scaled by sqrt(1536)
vision_encoder    (image[1,3,672,960] in [0,1])        → (1,280,1536)
audio_preprocessor (waveform[1,N_pcm])                 → (1,T,128)  dynamic T
audio_encoder     (mel[1,128,1976] channels-first)     → (1,494,1536)
```

PLI: text positions use real token IDs; image/audio positions use `pad_token_id=0`
(per HF `Gemma4Model.forward` line 2215). Decode steps use the current generated
token ID. All three are wired in `multimodal_prefiller.cpp` and
`Gemma4DecoderRunner::step()`.
